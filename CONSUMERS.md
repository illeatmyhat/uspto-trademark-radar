# Using this data in a detector

You don't query this data live and you don't run a database. It works like an
ad blocker: **we publish small filter lists; your extension fetches them,
caches them, and does a membership test.** Three lists, refreshed monthly:

- **`tier1.txt`** — the blocklist. Marks on a filing the **USPTO itself
  sanctioned**. A public-record fact, not our guess.
- **`tier2.txt`** — screened candidates. Marks whose filing features clear a
  recommended threshold. A hint to look closer, not a verdict.
- **`tier3.txt`** — marks filed by a **large, high-churn filing operation** — a
  structural fact about filing volume, keyed on the filing, naming no one. A
  stronger prior to inspect (it separates USPTO sanctions at ~6x the base
  rate), still not a verdict.

All three are hosted here (plain text, one mark key per line):

```
https://huggingface.co/datasets/illeatmyhat/uspto-trademarks-detector/resolve/main/tier1.txt
https://huggingface.co/datasets/illeatmyhat/uspto-trademarks-detector/resolve/main/tier2.txt
https://huggingface.co/datasets/illeatmyhat/uspto-trademarks-detector/resolve/main/tier3.txt
```

**One rule: match a mark, never a person.** These lists key on the mark text.
Do not build a blacklist of attorneys, agents, or applicants — see the bottom.

## How to use it

Fetch on install, store in IndexedDB, refresh monthly. At runtime, normalize
the brand text on the page and test membership — that's the whole thing.

```js
// 1. fetch + cache (do this on install and once a month)
const load = async (tier) => new Set(
  (await (await fetch(`https://huggingface.co/datasets/illeatmyhat/uspto-trademarks-detector/resolve/main/${tier}.txt`)).text())
    .split("\n").filter(Boolean));
const tier1 = await load("tier1");   // adjudicated blocklist
const tier2 = await load("tier2");   // screened candidates
const tier3 = await load("tier3");   // large-operation membership

// 2. normalize page text the SAME way the lists are keyed:
//    drop everything that isn't a-z / A-Z / 0-9, then uppercase
const key = (s) => s.replace(/[^A-Za-z0-9]/g, "").toUpperCase();

// 3. check a listing's brand
const k = key(listingBrandText);
if (tier1.has(k))      showFlag(k);        // USPTO sanctioned a filing under this mark
else if (tier3.has(k)) showStrongHint(k);  // filed by a large, high-churn operation
else if (tier2.has(k)) showSoftHint(k);    // worth a look
```

`tier1.txt` is a few hundred KB — small enough to bundle in the extension or
drop into a bloom filter. The lists refresh monthly, so re-fetch on a timer.

## Two things to get right

- **A mark maps to many filings.** A hit means "some filing under this mark was
  flagged," not "this exact listing." So **link out to the USPTO's own record**
  (a search for the mark text) and let the user decide — don't state a verdict.
  Some keys are short or generic and will match unrelated brands; that's
  expected. Ship broad and let users **allowlist** false positives as an
  exception, the same way an ad blocker handles them — don't prune the list.
- **Never blacklist people.** Bad actors forge and steal attorney credentials,
  so the name on a filing is often the *victim*. Names also collide, so a
  person list wrongly catches legitimate businesses, with no appeal. Match the
  mark; point at the USPTO record; leave judging people to the USPTO.

## Want to build the lists yourself, or tune the threshold?

The lists are generated by this repo, from the public corpus, with core
dependencies only:

```
trademark-radar gold              # build features + the local operation grouping
trademark-radar detector-lists    # -> data/gold/detector_v1/{tier1,tier2,tier3}.txt
#   --threshold 1.9               # raise it for a stricter, smaller tier-2 (default 1.8)
#   --min-cluster 1000 --min-churn 0.6   # tier-3 operation gate (defaults)
trademark-radar publish-detector --repo you/uspto-trademarks-detector
```

Tier 1 comes straight from USPTO sanctions events and status codes; Tier 2 is a
fixed, transparent blend of the published `mark_features` columns. Tier 3 is a
membership fact — a mark filed by an operation above a size and owner-churn gate
(`docs/adr/0017`); the operation grouping stays local, only the mark keys ship,
and owner nationality is deliberately not part of the gate. See `docs/ANALYSIS.md`.
